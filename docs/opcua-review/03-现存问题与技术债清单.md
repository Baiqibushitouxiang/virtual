# 现存问题与技术债清单

## 1. 总览

下面这些问题里，有些属于“功能没做完”，有些属于“设计上先省事，后面要补账”。我按优先级排序。

## 2. 高优先级问题

## 2.1 安全策略配置和实际实现不一致

现象：

- `application.yml` 里声明了多个 `security-policies` 和 `security-modes`
- 但 `OpcUaServerService.parseSecurityPolicies()` 目前只返回 `SecurityPolicy.None`
- `createEndpointConfigurations()` 也没有按配置真正组合多个安全模式

影响：

- 配置看起来支持 `Basic256Sha256` 等策略，实际并没有真正暴露对应能力。
- 容易让测试方误判系统已具备加密和签名支持。

结论：

这是最值得优先修的“表里不一”问题。

## 2.2 证书体系没有真正接到 OPC UA Server 认证链路

虽然项目里有：

- `CertificateUtils`
- `DeviceCertificateValidator`
- 设备证书生成接口

但当前 `OpcUaServerConfig` 构建时没有看到完整的证书管理器、身份校验器、证书信任链配置接入。

影响：

- 证书相关代码存在，但更像“准备工作”。
- 还不能证明 OPC UA 连接层已经完成证书认证闭环。

## 2.3 设备证书生成后没有返回私钥，不适合真实客户端使用

`DeviceController` 生成的是设备证书 PEM 和 thumbprint，并保存到数据库。

但问题是：

- 私钥没有持久化给设备端
- 只保存证书本身，设备无法拿这个结果完成真正的客户端证书认证

这更像“给设备做一个登记指纹”，不是真正可下发的设备证书包。

## 2.4 WebSocket token 校验过于宽松

`DeviceAuthenticationService.validateToken()` 目前基本只判断：

- 设备存在
- token 非空

影响：

- 这不是有效的认证方案。
- 任意非空 token 都可能通过。

如果 WebSocket 被当成设备接入通道，这个风险是很直接的。

## 2.5 通用数据节点没有落库闭环

`Temperature` 写入会进入 `DeviceDataService.saveDeviceData()`。

但 `Data` 节点写入后当前只会：

- 更新在线状态
- 更新最后活跃时间

它没有：

- 通用遥测落库
- JSON 结构校验
- 字段拆分
- 历史记录归档

结果就是最通用的那个节点，反而业务价值比较弱。

## 3. 中优先级问题

## 3.1 节点删除不彻底

`DeviceNamespace.removeDeviceNode()` 目前只是从 `DeviceNodeStore` 中移除。

没有看到它同步从 Milo 的 `NodeManager` 真正删除：

- 设备文件夹节点
- Data/Status/Temperature 子节点
- 组织引用关系

影响：

- 运行期地址空间可能残留幽灵节点。
- REST 状态页未必能看到，但 OPC UA 浏览侧可能还在。

## 3.2 节点模型写死，扩展性弱

当前每台设备固定只有：

- `Data`
- `Status`
- `Temperature`

这意味着每新增一种测点，都要改 Java 代码。

对 demo 来说能接受，对平台化接入非常不友好。

## 3.3 订阅体系代码存在，但主链路没有闭环

项目里有：

- `DataChangeMonitor`
- `SubscriptionManager`
- `DataPushService`

但从当前装配关系看，只有 `DataPushService` 与 `WebSocketHandler` 做了基本注入。

没有看到：

- `DeviceAuthenticationService` 注入到 `DeviceDataWebSocketHandler`
- `DataChangeMonitor` 注入到 `DeviceDataWebSocketHandler`
- `DataPushService` 正式订阅 `DataChangeMonitor`

影响：

- 一部分“实时订阅/变化广播”能力像是预备件，还不是稳定主链路。

## 3.4 OPC UA 和 WebSocket 两套接入模型并存，但职责边界不够清楚

当前项目既保留了 WebSocket 设备通信能力，也引入了 OPC UA。

优点是兼容。
问题是：

- 哪条才是正式设备接入主路径，不够明确。
- 状态、鉴权、数据推送在两套通道之间存在重复和分叉。

## 3.5 状态判断依赖定时任务轮询

`DeviceStatusCheckTask` 每 10 秒扫一次设备表，根据 `lastSeenAt` 判离线。

这对 demo 没问题，但在规模变大后会面临：

- 定时扫表成本增加
- 状态变化粒度粗
- 多实例下状态一致性问题

## 4. 低到中优先级问题

## 4.1 文档和部分页面存在乱码/编码问题

现有几个文档和若干前端文本、日志文本有明显乱码现象。

这会直接影响：

- 项目交接
- 评审印象
- 后续维护

它不影响程序主逻辑，但会明显拖累“可用感”。

## 4.2 测试覆盖较浅

已有测试说明团队不是零测试状态，这是好事。

但从当前 OPC UA 相关测试来看，更多是：

- bean 初始化
- 基本服务行为

缺少更关键的集成验证：

- 节点创建/写入/删除
- OPC UA 客户端真实连通
- 安全策略生效验证
- 温度写入后数据库和前端联动验证

## 4.3 配置暴露敏感默认值

`application.yml` 中存在数据库与 COS 的默认敏感配置样例。

即使只是开发默认值，也建议尽快迁到环境变量或示例配置中。

## 5. 架构层面的真实判断

当前最大的技术债，不是某一行代码有 bug，而是下面这件事：

> 项目已经长出了“工业接入平台”的外形，但内部仍然主要是“为一个演示闭环服务的定制实现”。

这并不是坏事，很多系统都会经历这个阶段。

真正需要避免的是：外形已经按平台宣传，内部却还按 demo 假设运行。

## 6. 优先级建议

建议按这个顺序处理：

1. 安全配置与实际能力对齐
2. 通用数据节点落库与模型化
3. 节点生命周期管理
4. 接入链路统一化
5. 运维观测与测试补齐
