# 当前 Demo 可用性评估

## 1. 总判断

### 1.1 结论

当前 demo **够简单，也能用，但只在一个比较窄的使用范围内成立**。

更准确地说：

- 对老师、甲方、内部评审展示来说，够用了。
- 对开发联调和概念验证来说，够用了。
- 对“接一个温度设备、演示实时写入和状态管理”来说，够用了。
- 对“通用工业设备接入 demo”来说，不够。
- 对“后续要扩成可稳定交付的接入层”来说，还差几个关键台阶。

## 2. 为什么说它“够简单”

## 2.1 部署简单

当前只需要启动后端，OPC UA Server 就会跟着起来。

默认端口也很直接：

- HTTP: `9999`
- OPC UA: `4840`

对 demo 来说，这是明显加分项，因为没有额外网关、没有额外 broker、没有独立配置中心。

## 2.2 使用路径简单

一个最小演示流程是：

1. 启动后端
2. 创建设备
3. 同步设备到 OPC UA
4. 用脚本连接 `opc.tcp://localhost:4840`
5. 对 `Temperature` 节点写值
6. 在前端看状态与数据变化

这条路径对非 OPC UA 专家也能走通。

## 2.3 节点模型简单

每个设备只有 3 个核心节点：

- `Data`
- `Status`
- `Temperature`

这对新手非常友好。地址空间不会一上来就复杂到难以浏览。

## 3. 为什么说它“还不够通用”

## 3.1 真实业务测点太少

现在真正带业务闭环的，几乎只有 `Temperature`。

这意味着 demo 实际上更接近：

“基于 OPC UA 采集温度示例”

而不是：

“支持多类型设备、多测点、多数据模型的通用接入 demo”

## 3.2 `Data` 节点是字符串兜底，不是规范模型

`Data` 节点目前适合写 JSON 字符串，但它只是一个 `String` 变量。

问题在于：

- OPC UA 的优势之一是结构化、类型化、可浏览。
- 把复杂数据塞进字符串，会把 OPC UA 又退化成“文本管道”。
- 对外部 OPC UA 客户端而言，可发现性和互操作性都一般。

## 3.3 前端管理页偏“节点台账”，不是“接入控制台”

当前前端页能做的事主要是：

- 看服务状态
- 看节点列表
- 同步设备
- 手动创建/删除节点
- 查看节点上的文本数据

还缺少真正接入平台会需要的能力：

- 证书状态
- 安全策略状态
- 节点浏览树
- 实时订阅曲线
- 连接日志
- 异常告警
- 测点模板

## 4. 现在这个 demo 适合什么场景

## 4.1 适合

- 课程设计或毕业设计演示
- 数字孪生平台一期功能展示
- 内部 PoC
- 向非工业软件背景成员解释 OPC UA 接入基本思路
- 演示“设备 -> 平台 -> 前端”的最小闭环

## 4.2 不适合

- 现场 PLC/网关批量接入
- 强安全要求环境
- 需要证书双向认证验收的场景
- 需要几十种测点模型的场景
- 需要运维追踪和故障定位的场景

## 5. 以“简单好用”为标准，当前得分如何

我把“简单好用”拆成 5 个维度：

| 维度 | 评价 | 说明 |
|---|---|---|
| 启动复杂度 | 高分 | 单体应用内嵌 OPC UA，启动很省事 |
| 演示闭环 | 中高分 | 温度写入链路完整 |
| 工业通用性 | 低分 | 数据模型过于单薄 |
| 安全完整性 | 低分 | 有配置外观，但实际生效不足 |
| 可维护性 | 中等偏低 | 目前更多靠代码约定，少统一抽象 |

## 6. 当前 demo 最适合的定义

建议对外这样定义，比较准确：

> 当前系统已实现一个可运行的 OPC UA 接入 demo，支持设备节点自动创建、基础状态管理、温度数据写入落库与前端展示，适合作为数字孪生平台的接入原型与后续扩展基础。

这个说法有三个好处：

1. 不会把当前能力说得过满。
2. 能突出“已经完成了可运行原型”。
3. 为二期优化留下合理空间。

## 7. 如果只想先把 demo 保持简单，最低限度该补什么

如果你们短期目标仍然是“保持 demo 轻量”，我建议至少补下面 5 项：

1. 让安全配置和实际生效一致。
2. 让 `Data` 节点至少支持基本 JSON 落库或字段拆解。
3. 删除节点时同步移除地址空间节点，而不只是删内存索引。
4. 给每个设备引入“测点列表”而不是写死 `Temperature`。
5. 把现有文档和脚本编码问题修掉，保证别人拿来能直接读。

这样改完以后，它仍然是 demo，但会从“能演示”变成“能放心拿给别人试”。
