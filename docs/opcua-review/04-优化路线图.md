# OPC UA 后续优化路线图

## 1. 优化目标

优化不建议一口气做成“全工业平台”，那样代价大、风险也高。更适合分三层推进：

1. 先把 demo 修成“可信 demo”
2. 再把接入能力做成“可扩展接入层”
3. 最后再考虑“可部署、可运维、可规模化”

## 2. 第一阶段：可信 demo

目标：不改变整体单体架构，先把最容易误导人的问题补平。

### 2.1 安全能力对齐

要做的事：

1. 让 `security-policies` 和 `security-modes` 配置真正参与 endpoint 构建
2. 明确当前支持哪些组合，哪些不支持
3. 如果暂时只支持 `None`，就把配置和文档都收窄，不要写出超出实际能力的描述

产出标准：

- 配置写什么，服务就真支持什么
- 管理端能看到当前实际开放的 endpoint 列表

### 2.2 证书链路最小闭环

要做的事：

1. 服务端证书加载和生成逻辑接入 OPC UA Server 配置
2. 设备证书生成时提供可用的客户端证书包导出方式
3. 区分“设备登记证书”和“设备连接证书”

产出标准：

- 至少能完成一条基于证书的真实连接验证链路

### 2.3 修正节点删除逻辑

要做的事：

1. 删除 `DeviceNodeStore` 记录
2. 删除 `NodeManager` 中的设备节点和子节点
3. 删除组织引用
4. 明确删除是否同步影响数据库设备主数据

产出标准：

- 删除后从 REST 状态页和 OPC UA 浏览器都看不到残留节点

### 2.4 文档与编码修复

要做的事：

1. 统一后端源码、文档、脚本、前端文案编码
2. 替换乱码日志和乱码界面文案
3. 用一个准确版本替代旧的 OPC UA 使用说明

产出标准：

- 新成员拿到仓库，文档能直接读，脚本说明不需要猜

## 3. 第二阶段：可扩展接入层

目标：从“温度样例”升级到“通用测点模型”。

### 3.1 把设备节点从写死模型改成模板模型

建议做法：

- 为设备增加“测点模板”或“点位定义”
- 每个设备按模板动态生成变量节点
- 测点属性至少包含：
  - pointCode
  - pointName
  - dataType
  - unit
  - rw 权限
  - 是否落库
  - 是否告警

建议地址空间模型：

```text
Objects
└── Devices
    └── {deviceId}
        ├── Meta
        │   ├── Status
        │   └── LastSeenAt
        └── Points
            ├── temperature
            ├── pressure
            ├── flow
            └── ...
```

这样做的价值：

- 新增测点不再改 Java 类
- 更接近工业设备点位建模习惯
- 前端和告警规则都更容易复用

### 3.2 通用写入处理管线

建议把当前写入处理统一成：

```text
节点写入
  -> 设备鉴权
  -> 点位识别
  -> 类型校验
  -> 可选换算/清洗
  -> 落库
  -> 规则评估
  -> 事件发布
  -> 前端推送
```

这样可以把 `Temperature` 特判慢慢收掉，避免后面每个测点都复制一套。

### 3.3 明确 OPC UA 与 WebSocket 的角色

建议二选一为主，另一条为辅：

方案 A：

- OPC UA 作为设备接入主协议
- WebSocket 只做前端实时展示

方案 B：

- WebSocket 继续服务轻量设备
- OPC UA 服务 PLC/工业网关

无论选哪种，都建议在文档和代码结构里写清楚，不要两条链路都像“正式入口”。

## 4. 第三阶段：可部署与可运维

目标：让它能承受更多设备、更长时间运行。

### 4.1 可观测性

建议补充：

- OPC UA 当前连接数
- 活跃设备数
- 每分钟写入量
- 节点创建/删除次数
- 写入失败次数
- 认证失败次数
- 告警触发次数

最好输出到：

- 管理页概览
- 应用日志
- 指标系统

### 4.2 状态管理优化

现在基于 `lastSeenAt + 定时扫描` 判离线。

后续可升级为：

- 写入即心跳
- 会话断开事件驱动
- 定时扫描兜底

这样状态会更准，也更节省资源。

### 4.3 多实例与独立化准备

如果未来设备数上来，建议把 OPC UA 接入层逐步拆为独立服务：

- `platform-api` 负责业务管理
- `opcua-gateway` 负责 OPC UA 会话和地址空间
- 两者通过消息或事件同步设备主数据

但这一步不建议太早做。当前先把单体版跑稳更值。

## 5. 推荐实施顺序

如果只给一个务实顺序，我建议：

1. 修安全配置与实际能力不一致
2. 修节点删除与证书闭环
3. 把 `Temperature` 特例推广成“通用测点写入”
4. 补管理端的证书/endpoint/连接状态可视化
5. 补真实 OPC UA 集成测试
6. 再评估是否拆分独立网关

## 6. 你们当前最划算的一步

如果时间有限，最划算的不是“大重构”，而是先做一个“小二期”：

- 保持单体部署
- 保持 Milo 内嵌
- 但把数据点模板、安全能力、节点生命周期补完整

这样能用最小成本，把项目从“演示 demo”推到“可认真试用的原型平台”。

## 7. 最终建议

短期定位建议：

> 把当前方案定义为“数字孪生平台的 OPC UA 接入原型”，先把原型做扎实，再决定是否演进为独立接入网关。

这是最稳，也最符合你们现有代码基础的路线。
