# 一期业务逻辑文档

## 1. 文档目标
- 面向实施、测试、产品和后续开发说明一期改造后的核心业务对象、操作流程与系统行为。
- 本文重点解释“设备、数据、模型、展板、告警”之间的关系，以及系统如何形成完整闭环。

## 2. 核心对象关系

### 2.1 设备
- 表示真实世界中的采集设备或控制设备。
- 负责身份、状态、证书、在线时间、所属用户等基础信息。
- 设备通过 `deviceId` 作为数据上报与告警匹配的核心编码。

### 2.2 设备数据
- 表示设备上报的某一条实时或历史采样值。
- 最关键字段：`deviceId`、`dataType`、`value`、`unit`、`recordedAt`。
- 设备数据是展板展示与告警判断的原始输入。

### 2.3 场景
- 表示一个数字孪生可视化场景资源。
- 场景负责模型布局、背景、纹理缓存与模型绑定元数据。
- 场景本体资源以 `json / glb / gltf` 存储，元数据保存在 `scene_assets.texture_info` 中。

### 2.4 模型
- 场景中的单个三维对象。
- 模型本身不直接保存设备数据，而是通过“模型数据绑定”与设备数据建立关系。

### 2.5 模型数据绑定
- 这是本轮改造新增的关键对象。
- 负责描述：某个场景里的某个模型，绑定到了哪个设备、哪个数据项。
- 典型字段：`sceneId`、`modelId`、`deviceId`、`deviceCode`、`dataType`。
- 这是“模型选择数据”的核心落点。

### 2.6 展板
- 展板现在只负责展示，不再承担主数据绑定入口。
- 它只需要知道“自己要跟哪个模型关联展示”。
- 真正的数据源来自该模型对应的“模型数据绑定”。

### 2.7 告警规则
- 表示针对 `设备 + 数据项` 的阈值判断配置。
- 设备数据写入后，系统根据规则执行阈值判断、冷却时间控制和通知输出。

## 3. 用户操作主路径

### 3.1 创建设备
- 输入：设备名称、描述等基础信息。
- 系统行为：生成设备编码、保存设备记录、可选同步到 OPC UA。
- 输出：可用于上报数据和绑定模型的设备档案。

### 3.2 验证数据接入
- 输入：设备通过接口或 OPC UA 写入实时数据。
- 系统行为：
  - 保存设备数据
  - 更新设备在线状态
  - 推送 WebSocket 数据
  - 触发告警规则评估
- 输出：实时数据可在监控页面看到，后续可被模型和展板使用。

### 3.3 配置场景与资源
- 输入：上传或打开场景资源、导入/导出 JSON、加载 GLB/GLTF。
- 系统行为：
  - 恢复模型列表
  - 恢复天空盒状态
  - 恢复纹理缓存
  - 恢复模型绑定元数据
- 输出：可编辑的数字孪生场景。

### 3.4 给模型绑定数据
- 输入：在场景编辑器中选中模型，再选择设备与数据项。
- 系统行为：保存一条模型数据绑定记录。
- 输出：该模型获得业务数据来源。

### 3.5 配置展板
- 输入：新增展板、绑定展板到模型、调整展板位置。
- 系统行为：展板根据“展板绑定模型 + 模型绑定数据源”进行展示。
- 输出：用户看到模型对应的实时数据浮窗。

### 3.6 设置告警
- 输入：给 `设备 + 数据项` 设置阈值规则。
- 系统行为：当设备数据写入时即刻评估是否命中规则。
- 输出：命中时执行控制台告警输出。

## 4. 场景导入导出逻辑

### 4.1 导出逻辑
- 输入：当前编辑器中的场景对象。
- 系统行为：
  - 过滤掉 helper、灯光等运行时对象
  - 序列化模型 `id/uuid/name/file/position/rotation/scale/userData`
  - 序列化天空盒状态
  - 序列化模型绑定列表
- 输出：版本化场景 JSON。

### 4.2 导入逻辑
- 输入：JSON 文件或场景库中的场景资源。
- 系统行为：
  - 新格式直接按 `version/sceneMeta/models/skybox/bindings` 读取
  - 旧格式按兼容规则补默认值
- 输出：场景、模型、背景和绑定关系恢复到编辑器。

### 4.3 当前 v1 限制
- 场景元数据复用了 `texture_info` 字段，不是独立表结构。
- GLB 场景保存与 JSON 元数据是并行关系，需要联调时重点确认恢复链路。

## 5. 模型数据绑定逻辑

### 5.1 输入
- `sceneId`
- `modelId`
- `modelName`
- `deviceId`
- `dataType`

### 5.2 系统行为
- 若同一 `sceneId + modelId` 已存在绑定，则更新原记录。
- 若不存在，则新建绑定。
- 同时根据 `deviceId` 回填 `deviceCode` 与 `deviceName`。

### 5.3 输出
- 一条可被展板与场景元数据共同消费的绑定记录。

### 5.4 当前 v1 限制
- 数据项类型是约定值，如 `temperature / humidity / pressure / voltage / current / power`。
- 未引入独立字典表，前后端需保持枚举一致。

## 6. 展板展示逻辑

### 6.1 输入
- 展板绑定到模型
- 模型绑定到设备与数据项
- WebSocket 推送的实时设备数据

### 6.2 系统行为
- 展板优先根据自身绑定的模型找到对应 `modelBinding`
- 从 `modelBinding` 中得到 `deviceCode + dataType`
- 只展示与该模型绑定数据源匹配的 WebSocket 消息

### 6.3 输出
- 模型对应的数据浮窗展示
- 展板拖拽位置持久化

### 6.4 当前 v1 限制
- 展板仍保留旧模型绑定接口，用于确定展板跟随哪个模型。
- 主数据绑定不在展板页完成，而在模型详情页完成。

## 7. 告警触发逻辑

### 7.1 输入
- 设备数据写入请求
- 告警规则：设备 + 数据项 + 运算符 + 阈值 + 冷却时间

### 7.2 系统行为
- 数据保存成功后立即查询匹配规则
- 对每条规则执行运算符比较：`>` `>=` `<` `<=`
- 若命中且不在冷却期内：
  - 生成消息文本
  - 执行通知适配器
  - 回写 `lastTriggeredAt`
- 若在冷却期内：跳过重复通知

### 7.3 输出
- 当前 v1 为控制台打印告警日志
- 后续可替换为外部 HTTP、短信、企业微信等通知方式

### 7.4 当前 v1 限制
- 通知适配层已预留，但仅实现 `ConsoleAlertNotifier`
- 未实现告警历史表与消息发送回执存储

## 8. 实施与测试关注点
- 先验证设备数据是否真的进入系统，再验证模型绑定与展板展示。
- 场景 JSON 新旧格式要分别测试。
- 告警规则要重点测试四种运算符和冷却时间。
- 若本地环境启用 COS 存储，联调前需确认 COS 配置有效；若仅做本地测试，建议切到 `local` 存储模式。

## 9. v1 约束总结
- 告警通知仅控制台打印。
- 场景元数据暂存于 `texture_info`。
- 模型数据项为固定枚举，不支持动态字典配置。
- 联调依赖本地 MySQL `dam`、后端服务可用、必要的模型与场景资源存在。
